작업 지시사항:

언어 규칙 (매우 중요):

- 모든 대화 응답뿐 아니라,
  작업 계획, Task List, Progress Updates,
  Implementation Plan, Summary, Review 문서,
  설계 문서, 주석이 아닌 설명 문서는
  반드시 한국어로 작성한다.

- 기술 문서, 설계 문서, 작업 계획이라고 하더라도
  영어로 작성하지 않는다.

- 기존에 영어로 작성된 내용이 있더라도,
  이후 생성되는 모든 산출물은 한국어로 작성한다.

1. 저장소 및 작업 환경
- 본 작업은 로컬 환경에서 수행됨을 전제로 한다.
- 저장소는 최신 상태라고 가정하되, 필요 시 최신 상태 여부를 검토한다.
- 실제 커밋, 푸시, 배포는 사용자가 직접 수행한다.

2. 작업 범위 및 역할
- 사용자는 개선사항 요청뿐 아니라, 다음과 같은 요청을 할 수 있다:
  - 코드 품질 평가
  - 구조적 문제 분석
  - 개선 방안 또는 대안 설계 제안
- 수정이 필요한 경우, 먼저 변경 방향과 내용을 설명한 뒤 작업을 진행한다.

3. 코드 변경
- 수정 또는 개선 작업은 로컬 파일 변경 기준으로 수행한다.
- 변경 사항은 명확히 식별 가능하도록 정리한다.
- 모든 변경은 단계적으로 진행하며, 중간 결과를 설명한다.

4. 테스트 및 검증
- 변경 후 가능한 범위 내에서 빌드, 실행, 적용 여부를 확인한다.
- 오류가 발생할 경우:
  - 오류 원인을 분석한다.
  - 수정 방안을 제시하고 반영한다.
  - 재확인 과정을 반복한다.
- 테스트 범위는 현실적으로 실행 가능한 수준으로 제한한다.

5. Git / PR 정책
- Pull Request 생성은 기본적으로 수행하지 않는다.
- PR 생성은 사용자가 명시적으로 요청한 경우에만 수행한다.
- 브랜치 전략, 커밋, 푸시는 사용자가 직접 관리한다.

6. 배포 구조 인지 (중요)
- 이 저장소의 코드는 그대로 빌드되거나 배포되지 않는다.
- 폴더별 배포 구조는 다음과 같다:
  - backend/ : 폴더 내 모든 파일이 별도의 Git 저장소로 업로드되어 Render로 배포된다.
  - db/ : 폴더 내 파일을 기반으로 Neon DB에서 구축된다.
  - ui/ : 폴더 내 모든 파일이 GitHub Pages로 배포된다.

7. DB 관련 규칙
- DB 변경 시 schema 변경만 하지 않는다.
- 이미 구축된 DB에 반영하기 위한 migration SQL을 반드시 작성한다.
- migration 파일은 migrations/ 폴더에 추가로 생성한다.
- migration에는 적용 순서 및 롤백 가능 여부를 고려한다.

8. 이후 제공되는 요청사항(개선, 평가, 질문 등)을 기준으로 작업을 진행한다.

추가 규칙: 코드 일관성 및 변경 제한 (중요)

- 기존 코드 스타일, 아키텍처, 폴더 구조, 네이밍 규칙을 최대한 유지한다.
- 개선 작업은 “문제 해결에 필요한 최소 변경(minimal diff)” 원칙으로 진행한다.
- 대규모 리팩터링(구조 전면 변경, 패턴 교체, 프레임워크/라이브러리 교체)은 사용자가 요청한 대안 제시 등을 통한 명시적 요청이 있지 않는 한 제한한다.
- 파일/폴더 이동, 대량의 포맷팅 변경(전역 prettier/black 적용 등), 광범위한 rename은 제한한다.

프론트(UI) 관련 추가 제한:
- UI 레이아웃, 디자인, 스타일(색/간격/폰트/컴포넌트 구성)을 임의로 변경하지 않는다.
- HTML 구조를 유지한다(필요 시 최소한의 요소/속성/클래스 추가만 허용).
- CSS/스타일 파일의 전면 수정, 클래스명 대량 변경, 디자인 전반에 영향을 주는 변경은 제한한다.
- UI 변경이 불가피한 경우, 변경 이유/대안/영향 범위를 먼저 제안하고 사용자의 명시적 승인을 받은 후에만 적용한다.

작업 절차:
- 코드 변경이 필요한 경우, 먼저 변경 계획(어떤 파일을 왜 얼마나 바꾸는지)을 요약하고 승인을 요청한다.
- 승인 전에는 실제 파일 수정/명령 실행을 하지 않는다(Review-driven 기준).

추가 규칙: 비용/리소스 절감 및 점진적 개선 (매우 중요)

- Render 및 Neon DB는 무료 플랜을 사용하므로, 사용량(빌드/배포 횟수, 실행 시간, Neon CU/쿼리 부하 등)을 과도하게 소비하지 않는 방향으로 개발한다.
- 변경은 “작게, 자주 검증” 원칙으로 진행하되, 불필요한 재배포/재시작/반복적인 무거운 테스트 실행을 유발하는 방식은 피한다.
- DB는 비용/부하가 큰 작업(대량 데이터 갱신, 전체 테이블 재작성, 인덱스 난립, 고비용 쿼리)을 기본적으로 피하고, 필요한 경우에도 대안을 함께 제시한다.
- DB 스키마 변경은 최소화한다. 새로운 컬럼/테이블 추가가 필요할 때는:
  1) 기존 구조 내 해결 가능한지 먼저 검토하고,
  2) 불가피할 경우에만 진행하며,
  3) migration은 가능한 “가벼운 변경”으로 작성한다.
- Neon에 부담이 큰 쿼리(풀스캔, N+1, 큰 JSON 다량 저장/조회 등)가 예상되면, 사전에 위험을 경고하고 더 가벼운 대안을 제시한다.
- Render 배포를 전제로 한 변경(의존성 추가, 빌드 단계 변경, 런타임 변경)은 특히 신중히 하며, 꼭 필요할 때만 제안한다.

추가 규칙: 스코프 관리 및 과한 변경 방지 (보편 규칙)

- 요청사항의 크기와 무관하게, 한 번에 전면 개편하지 않고 단계적으로 제안한다.
- “기존 동작/구조/스타일을 유지하면서 효과가 큰 최소 변경안”을 항상 1순위로 제시한다.
- 다음 유형의 변경은 기본적으로 금지하며, 사용자의 명시적 승인(‘이 방향으로 진행’ 등)이 있을 때만 수행한다:
  - 대규모 리팩터링(구조 전면 변경, 패턴/프레임워크 교체, 광범위한 파일/폴더 이동)
  - 데이터 모델의 근본 변경(새 테이블 대량 추가, 핵심 스키마 재설계, 저장 포맷 전환)
  - UI/UX 전면 개편(레이아웃/플로우 전반 변경, 편집기/화면 구조 교체)

- 계획(Implementation Plan) 작성 시 반드시 다음을 포함한다:
  1) 1단계 최소 변경안: 기존 구조를 유지하며 리스크가 낮은 개선
  2) 2단계 확장안(선택): 필요 시 일부 구조 개선을 포함한 개선
  3) 3단계 고급안(선택): 사용자 경험을 크게 바꾸는 기능/인터랙션 추가(필요할 때만)
  각 단계별로 변경 범위(대상 파일/모듈), 기대효과, 리스크/비용(개발/테스트/운영)을 짧게 비교한다.

- 구현은 1단계부터 진행하며, 2~3단계는 사용자의 별도 승인 없이는 착수하지 않는다.
